The 5 Building Blocks

Every good prompt answers five questions

1 Role Who is the AI?
"Senior React engineer"
2 Context What's the stack?
"TS, Tailwind, Next.js"
3 Constraints What NOT to do?
"No any, no defaults"
4 Format What shape?
"Component + barrel export"
5 Examples Show good & bad
"Here's a pattern to follow"

You don't need all five every time. But when output isn't right, check which one you're missing.

Quick walkthrough. Don't linger — the audience needs the framework, but the demo is what proves it. Point to each pillar and give the one-liner. "Think of these as controls on a mixing board."

Constraints: The Secret Weapon

The most underused and highest-leverage lever

Do NOT:
- Use default exports
- Use inline styles or CSS modules
- Use `any` type or type assertions
- Add comments explaining basic concepts
- Add console.log statements
- Use class components

Telling the model what NOT to do narrows the output space faster than describing what you want.

Think of a creative brief — "two colors max, no gradients, must work at 16px" gets you focused work.
"Make it pop" gets you chaos.

This is the single biggest unlock for most people. When your output isn't right, don't rewrite the whole prompt — just add constraints. The "Do NOT" block is as important as the requirements block.

The APE Framework

Action → Purpose → Expectation

Vague

Fix this function, it's timing out

Model adds limit: 50. Problem "fixed." 450 products silently dropped.

APE Prompt

Action Refactor fetchProducts to use cursor-based pagination
Purpose 500+ product collections cause API timeouts at 30s limit
Expectation Batches of 100, loading callback, abort support, graceful degradation

Three seconds of thought. Three fewer retries.

APE is the easiest framework to remember. Action = what. Purpose = why. Expectation = done. The Purpose is the key — when the model knows WHY, it makes better architectural decisions. Don't linger here. Quick hit, move to the workflow slide.

Building a Project

Three messages, one thread. Each builds on the last.
The model remembers everything.

Message 1: Core

ProductReviewCard with star rating, review text, reviewer name + date

Message 2: Interactivity

Helpful / Not Helpful buttons, optimistic UI, toggle behavior

Message 3: Accessibility

Screen reader announcements, aria-pressed, keyboard nav, aria-live

Three focused messages beat one mega-prompt and three separate conversations.

Switch to VS Code. Stay in the same Copilot Chat thread. Key point: you don't re-specify TypeScript, Tailwind, etc. — it already knows. Message 2: "Watch — it modifies the existing component, not a new one." Message 3: "Each concern gets 100% attention." If tight on time, skip Message 3 and describe what it would do.

copilot-instructions.md

A persistent system prompt. Write it once. The whole team benefits.

# .github/copilot-instructions.md

## Stack
- React 18 + TypeScript 5.4 (strict mode)
- Next.js 14 App Router (server components by default)
- Tailwind CSS 3.4 (no CSS modules, no styled-components)
- Zod for runtime validation at API boundaries

## Code Rules
- `const` only, arrow functions exclusively
- Named exports only (no default exports)
- No `any` — use `unknown` and narrow with type guards
- No inline styles, no explanatory comments
- Props destructured on their own line

## Naming
- Files/folders: kebab-case (product-card.tsx)
- Components: PascalCase, Hooks: usePrefix

## File Structure
  my-component/
    my-component.tsx     # component
    my-component.test.tsx
    index.ts             # barrel export

New developer joins → AI output matches your team's standards from day one.

This is the scalability play. Lives in .github/, version-controlled, reviewed like any other code. It's a system prompt for your whole team. 30 minutes to set up. "This is how you go from 'AI helps individuals' to 'AI enforces team standards.'"

Key Takeaways

1
Engineer the prompt, don't just ask Prompts are specifications, not questions. Structure them like requirements.
2

Iterate in one thread, don't restart Context accumulates. Three focused messages beat three separate conversations.
